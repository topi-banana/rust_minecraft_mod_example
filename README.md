# rust_minecraft_mod_example

[日本語版 README](./README_jp.md)

[![CI](https://github.com/topi-banana/rust_minecraft_mod_example/actions/workflows/ci.yml/badge.svg)](https://github.com/topi-banana/rust_minecraft_mod_example/actions/workflows/ci.yml)

A Fabric Minecraft mod toolchain that builds `.class` files and the mod
jar entirely from Rust — no Java toolchain, no Gradle, no Mixin Gradle
plugin. Server-side hooks are written as small `wasm32-unknown-unknown`
binaries annotated with a single `#[inject(...)]` attribute; the host
side uses [wasmer-java][wj] to load and call them at runtime.

[wj]: https://github.com/wasmerio/wasmer-java

## How it works

```
crates/wasm-payloads/src/bin/hello.rs
  ┌────────────────────────────────────────────────────────────┐
  │ #[inject(target = "Lnet/.../MinecraftServer;",             │
  │         method = "runServer", at = "HEAD")]                │
  │ pub extern "C" fn hello() -> u64 { ... }                   │
  └────────────────────────────────────────────────────────────┘
                              │
              cargo build --target wasm32-unknown-unknown
                              ▼
   target/.../release/hello.wasm  +  custom section "inject_meta"
                              │
                       cargo run -p builder
                              ▼
   out/hello-wasm-mod.jar
     ├─ fabric.mod.json
     ├─ hello-wasm-mod.mixins.json
     ├─ com/example/mixin/hello_hello_Mixin.class   ← crustf-generated bytecode
     ├─ assets/hello-wasm-mod/hello.wasm
     └─ org/wasmer/**                                ← flattened wasmer-jni
```

1. The `#[inject]` proc macro records `(fn_name, target, method, at)`
   into a wasm custom section named `inject_meta`.
2. The host bin (`builder`) uses [`cargo_metadata`] to enumerate
   `wasm-payloads`' bins, [`wasmparser`] to read each wasm's
   `inject_meta`, and [`crustf`] to emit one parameterized Mixin class
   per record.
3. The Mixin class loads its bundled `.wasm` resource at runtime,
   instantiates it via wasmer-java, calls the exported function, reads a
   string back through linear memory, and prints it.

## Prerequisites

- Rust 1.95+
- `wasm32-unknown-unknown` target — `rustup target add wasm32-unknown-unknown`
- `wasmer-jni-{platform}-0.3.0.jar` from
  [wasmer-java releases][wj-releases] placed under `vendor/`. amd64
  Linux / macOS / Windows are supported. Drop in as many platform jars
  as you have access to; the builder merges them so a single mod jar
  covers every platform whose lib is present.
- A Minecraft + [Fabric Loader] 0.15+ install on Minecraft 1.20+ for the
  end-to-end test.

[wj-releases]: https://github.com/wasmerio/wasmer-java/releases
[Fabric Loader]: https://fabricmc.net/

## Build

```
cargo run -p builder
```

Produces `out/hello-wasm-mod.jar`. Drop into `<minecraft>/mods/`. On
server start, the Mixin injects at `MinecraftServer#runServer @At("HEAD")`
and you should see `Hello from WASM!` in the log.

## Adding a hook

1. Create `crates/wasm-payloads/src/bin/<name>.rs`.
2. Annotate one or more exports with `#[inject(...)]`.
3. `cargo run -p builder` picks it up automatically and adds another
   Mixin class to the jar.

```rust
#![cfg_attr(target_arch = "wasm32", no_main, no_std)]

#[cfg(not(target_arch = "wasm32"))]
compile_error!("wasm-payloads must be built with `--target wasm32-unknown-unknown`");

#[cfg(not(target_arch = "wasm32"))]
fn main() {}

const MSG: &str = "another hook!";

#[inject_macro::inject(
    target = "Lnet/minecraft/server/MinecraftServer;",
    method = "loadWorld",
    at = "TAIL"
)]
pub extern "C" fn run() -> u64 {
    let p = MSG.as_ptr() as u32 as u64;
    let l = MSG.len() as u32 as u64;
    (p << 32) | l
}

#[cfg(target_arch = "wasm32")]
#[panic_handler]
fn ph(_: &core::panic::PanicInfo) -> ! {
    core::arch::wasm32::unreachable()
}
```

## `#[inject]` contract

| field    | required                | meaning                                                            |
|----------|-------------------------|--------------------------------------------------------------------|
| `target` | yes                     | JVM internal-form descriptor of the class to mix into              |
| `method` | yes                     | name of the method on the target class                             |
| `at`     | no — defaults to `HEAD` | Mixin `@At` injection point: `HEAD`, `TAIL`, `RETURN`, etc.        |

The annotated function **must** be `pub extern "C"` and currently
**must** have the signature `() -> u64` returning `(ptr << 32) | len`
for a UTF-8 string in linear memory. The host reads `len` bytes from
`ptr` via `ByteBuffer` and prints them with `System.out.println`.

## Workspace layout

```
crates/
├── inject-macro/      proc-macro for #[inject(target, method, at)]
├── wasm-payloads/     bins built for wasm32-unknown-unknown only
└── builder/           host bin that emits the mod jar via crustf
```

`wasm-payloads` is excluded from workspace `default-members` and
contains a `compile_error!` guard that rejects non-wasm32 targets, so
running `cargo build` from the workspace root only touches the host
crates. The builder invokes
`cargo build -p wasm-payloads --target wasm32-unknown-unknown` itself
when needed.

## Limitations

- One signature shape: `pub extern "C" fn() -> u64` returning packed
  `(ptr, len)` for a UTF-8 string. Other signatures are not supported in
  v1.
- `compatibilityLevel: JAVA_8` on the Mixin (class file version 52);
  the generated `onRun` body is branchless so no `StackMapTable` is
  needed.
- wasmer-java 0.3.0 is dormant upstream and amd64-only. ARM (Apple
  Silicon, ARM Linux) is not covered.
- The mod jar carries every platform's native lib (≈5–10 MB each), so
  size scales linearly with the platforms you bundle.
